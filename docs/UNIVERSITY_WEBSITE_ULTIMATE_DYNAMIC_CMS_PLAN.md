# خطة ترقية موقع الجامعة إلى نسخة Dynamic CMS شاملة

> النطاق: هذه الخطة خاصة بموقع الجامعة الإلكتروني الرسمي التعريفي فقط (Official Electronic University Website)، وليست LMS.

## 1. الملخص التنفيذي

الموقع وصل إلى جاهزية إنتاجية جيدة، والخطوة التالية هي تحويله إلى منصة مؤسسية ديناميكية بالكامل من منظور إدارة المحتوى، بحيث تصبح أغلب عناصر الواجهة العامة قابلة للإدارة من لوحة التحكم بدون تدخل تطوير مستمر.

الخطة مقسمة إلى 5 مراحل متدرجة لتقليل المخاطر والحفاظ على الاستقرار:
- Phase 1: Dynamic Homepage + Global Identity
- Phase 2: Advanced Page/Section Builder
- Phase 3: Temporary Admissions Workflow Enhancement
- Phase 4: LMS/Payment Integration Readiness Layer
- Phase 5: Production Hardening + Tests + Audit Logs + Final QA

## 2. الهدف من النسخة الفتاكة

1. تحويل الموقع إلى حضور جامعي مؤسسي ديناميكي قابل للإدارة شبه الكاملة من الداشبورد.
2. الحفاظ على هوية الجامعة كجامعة إلكترونية رسمية تقدم برامج أكاديمية مرنة.
3. تجهيز بنية قابلة للتكامل لاحقًا مع LMS وبوابات الدفع بدون إدخال ميزات LMS الآن.
4. تقليل الاعتماد على التعديلات البرمجية لكل تحديث محتوى.

## 3. ما المقصود بأن الموقع كله Dynamic

المقصود أن أغلب ما يراه الزائر في الواجهة العامة يُدار من CMS، بما يشمل:
- عناوين ووصف الأقسام
- بطاقات وروابط الأقسام
- ترتيب العناصر وإظهار/إخفاء الأقسام
- بيانات الهوية العامة (اسم الجامعة، الشعار، الوصف، التواصل، السوشيال)
- إعدادات SEO الافتراضية العامة

مع الإبقاء على fallback آمن عند غياب المحتوى المنشور.

## 4. الوحدات التي يجب إدارتها من الداشبورد

1. إدارة هوية الموقع العامة (Global Identity)
2. إدارة أقسام الصفحة الرئيسية
3. إدارة عناصر/بطاقات الصفحة الرئيسية
4. إدارة صفحات الثقة المؤسسية (موجودة وتُستكمل)
5. إدارة المراكز والمجلات والاعتمادات والشهادات (موجودة وتُستكمل)
6. إدارة الفوتر والروابط العامة
7. إدارة SEO الافتراضي على مستوى الموقع

## 5. Phase 1: Dynamic Homepage and Global Identity

### الهدف
تحويل الصفحة الرئيسية إلى صفحة ديناميكية مُدارة من CMS مع الحفاظ على التصميم المعتمد وهوية الجامعة الإلكترونية.

### النطاق التنفيذي
1. نموذج بيانات منظم لأقسام الصفحة الرئيسية:
   - hero
   - quick_access
   - electronic_university_model
   - study_pathways
   - faculties_programs
   - trust
   - research_centers_journals
   - admissions_journey
   - news_events
   - final_cta
2. نموذج بيانات لعناصر الأقسام (cards/links/steps/buttons).
3. API عام يعرض المنشور والمفعل فقط.
4. API إداري محمي لإدارة الأقسام والعناصر.
5. واجهة إدارة داخل لوحة التحكم لإدارة Homepage Sections + Items.
6. ربط `Home.tsx` ببيانات CMS مع fallback آمن.
7. تحسين إعدادات الهوية العامة (الاسم، الشعار، الوصف المختصر، التواصل، السوشيال، SEO defaults).

### المخرجات
- Homepage ديناميكية بالكامل ضمن الهيكل المعتمد.
- عدم كسر RTL/LTR أو SEO الحالي.
- عدم كسر routes أو dashboard الحالية.

## 6. Phase 2: Advanced Page/Section Builder

### الهدف
توسعة إدارة المحتوى إلى Page/Section Builder شبه مرن للصفحات التعريفية (بدون تعقيد drag-and-drop كامل في البداية).

### نطاق مقترح
1. تعريف Section Templates قياسية (Hero, Rich text, Cards, Stats, CTA, FAQ block).
2. إدارة ترتيب sections على مستوى الصفحة.
3. دعم publish/draft ونسخ لغة عربية/إنجليزية لكل section.
4. ربط builder بصفحات محتوى مؤسسية إضافية.

## 7. Phase 3: Temporary Admissions Workflow

### الهدف
تحسين مسار القبول الحالي كنظام مؤقت قبل تكامل الدفع وLMS.

### نطاق مقترح
1. حالات طلب القبول بشكل أوضح:
   - pending_review / accepted_conditionally / accepted / rejected / waitlisted
2. حالات الدفع المؤقتة:
   - not_required
   - pending_payment
   - payment_under_review
   - paid_manual
   - failed
   - gateway_pending_future
3. حالات جاهزية تكامل LMS:
   - not_ready
   - pending_lms_integration
   - ready_to_sync
   - synced
   - sync_failed
4. حقول ملاحظات الإدارة + معرفات تكامل مستقبلية (paymentReference / lmsUserId / lmsEnrollmentId).

> ملاحظة: لا يتم تنفيذ بوابة دفع حقيقية أو LMS في هذه المرحلة.

## 8. Phase 4: LMS and Payment Integration Readiness

### الهدف
بناء طبقة تكامل جاهزة للتوصيل لاحقًا بدون تغيير جذري في الموقع.

### نطاق مقترح
1. Integration contracts واضحة (interfaces/events).
2. Webhook/event log داخلي لمحاولات التكامل.
3. Mapping layer بين طلبات القبول وكيانات التكامل الخارجية.
4. آلية retries وإدارة أخطاء تكامل مستقبلية.

> لا يتم إدخال ميزات LMS التشغيلية داخل الموقع التعريفي.

## 9. Phase 5: Production Hardening

### الهدف
رفع الاعتمادية التشغيلية قبل الإطلاق النهائي طويل المدى.

### نطاق مقترح
1. اختبارات smoke/regression أساسية آلية.
2. إضافة lint/test scripts وخط تحقق CI.
3. Audit logs للتعديلات الحرجة في CMS.
4. تحسينات Accessibility النهائية.
5. مراجعة أمنية نهائية للنماذج العامة.
6. مراجعة أداء نهائية (bundle/chunk/caching).

## 10. ما لن يتم إدخاله الآن لأنه يخص LMS

1. لوحة تعلم الطالب التعليمية.
2. لوحة المعلم التعليمية.
3. إدارة المقررات كمنصة تعليمية.
4. الاختبارات والواجبات والدرجات.
5. تتبع التقدم الدراسي داخل المقررات.
6. الفصول الافتراضية ومشغل الدروس.

## 11. المخاطر

1. تضخم Phase 1 إذا تم تنفيذ Page Builder مرن بالكامل مبكرًا.
2. إدخال محتوى غير معتمد رسميًا من الإدارة قد يؤثر على المصداقية.
3. تداخل إدارة الصفحة الرئيسية بين settings وhomepage modules بدون governance واضح.
4. نمو schema دون migration strategy رسمية قد يصعب الصيانة لاحقًا.

## 12. Acceptance Criteria

1. جميع أقسام الصفحة الرئيسية الأساسية قابلة للإدارة من الداشبورد (sections + items).
2. الصفحة الرئيسية تعرض بيانات CMS المنشورة عند توفرها وتعود إلى fallback آمن عند غيابها.
3. RTL/LTR سليم في العربية والإنجليزية بعد التحويل إلى dynamic content.
4. لا يوجد أي محتوى/مصطلحات LMS في الواجهة العامة.
5. أوامر التحقق (typecheck/build) تمر بنجاح.
6. لا يتم كسر لوحة الإدارة أو المسارات العامة الحالية.
7. توثيق ما تم تنفيذه في تقرير Phase 1 بشكل واضح.
